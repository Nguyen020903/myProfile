About

Problem-solving with measurable outcomes, not buzzwords.

I approach software as systems: boundaries between services, failure modes, and how data flows when traffic or scope grows. My background is in IT and full-stack development, with hands-on experience integrating language models and automation into existing products—always behind explicit contracts (REST, MCP tools, webhooks) so behavior stays testable and observable.

On the frontend I prioritize accessible markup, predictable state, and CSS that scales. On the backend I lean on Node.js and Express for fast iteration, MongoDB when document shapes evolve, and structured logging plus health checks before calling something “done.”

How I work

  • Define API contracts early; document edge cases.
  • Automate repetitive flows (CI, n8n, or small scripts) after the manual path is understood.
  • Ship thin vertical slices, then harden with monitoring and tests.

Projects

Selected work with explicit problems, constraints, and technical decisions.

Operations Console

Full stack

Problem

Support and ops teams were juggling three separate tools to view customer accounts, subscription state, and manual fixes—slowing response times and increasing error rates.

Solution

Built a single React dashboard backed by an Express service layer over MongoDB, with role-based views and audit-friendly action logs. Introduced a normalized read model for list screens and explicit write endpoints for sensitive operations.

Tech

React · Express · MongoDB · JWT sessions · REST

Highlights

  • Designed pagination and filter queries against compound indexes to keep list views fast on growing collections.
  • Implemented optimistic UI for low-risk edits with server reconciliation on failure.
  • Added structured error payloads so the UI could surface actionable messages without leaking internals.

Internal Docs MCP Server

MCP · Node

Problem

Engineers wanted LLM-assisted help inside the IDE, but public models had no access to private runbooks and API nuances—copy-pasting fragments was brittle and unsafe.

Solution

Shipped a Model Context Protocol server in Node.js that exposes curated tools: search indexed internal docs, fetch OpenAPI snippets by operationId, and return only redacted examples. Integrated with the team’s existing markdown repo and cache layer.

Tech

Node.js · MCP SDK · Express (health) · Redis-style in-memory cache (dev)

Highlights

  • Defined tool schemas with strict input validation to avoid ambiguous calls from agents.
  • Implemented RAG-style retrieval over chunked markdown with source citations in responses.
  • Documented transport setup (stdio vs SSE) for local and shared deployments.

Lead Routing Automation

n8n · APIs

Problem

Inbound leads from forms and a CRM webhook arrived in bursts; manual triage in a spreadsheet caused duplicate outreach and missed SLAs.

Solution

Designed an n8n workflow that normalizes payloads, deduplicates by email hash, scores urgency from form fields, and opens tickets via REST. Failures route to a dead-letter queue with replay instructions.

Tech

n8n · REST webhooks · Express middleware (signature verify) · MongoDB (idempotency keys)

Highlights

  • HMAC verification on inbound hooks before side effects.
  • Idempotent writes so retries from the workflow engine never double-book.
  • Runbook for rotating credentials and monitoring workflow execution time.

Public REST API & Partner SDK

API design

Problem

Partners needed stable programmatic access to inventory and order status, but internal services were built for synchronous web traffic only—no versioning, no rate limits.

Solution

Introduced versioned REST resources under /v1, API keys with scoped roles, and sliding-window rate limiting. Published OpenAPI and a minimal JavaScript SDK that wraps auth and pagination.

Tech

Express · MongoDB · express-rate-limit · OpenAPI 3 · Jest

Highlights

  • Consistent error envelope: code, message, requestId for support correlation.
  • Integration tests against a disposable MongoDB fixture for CI.
  • Deprecation headers and changelog for breaking vs additive changes.
